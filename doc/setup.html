Setup Guide

Deploy your own Cutout instance. Roughly half the steps are in the Cloudflare dashboard, half in code.

Prerequisites

  • A Cloudflare account with at least one domain on it
  • Nix installed (or Rust + wrangler + worker-build manually)

1. Clone and enter the dev shell

git clone https://github.com/ananthb/cutout.git
cd cutout
nix develop

2. Create the KV namespace

wrangler kv namespace create KV

Put the returned id into the [[kv_namespaces]] block of wrangler.toml.

3. Enable Cloudflare Email Routing (inbound) on each domain

In the dashboard for every email domain you want Cutout to handle:

  1. Open Email → Email Routing and click Enable. Accept the MX and SPF DNS records Cloudflare offers to add.
  2. Under Routes, edit the catch-all address, set the action to Send to a Worker, and pick the cutout worker. Save and enable the rule.

Cutout is domain-agnostic — one worker can serve any number of zones. Just repeat this step per domain.

4. Onboard each domain to Cloudflare Email Service (outbound)

The reverse-alias reply path sends mail to arbitrary original-sender addresses, which can't be pre-verified in Email Routing. Onboard each sender domain to Email Service:

  1. Dashboard → Email → Email Sending → Onboard Domain
  2. Accept the DNS records (MX / SPF / DKIM / DMARC) on the cf-bounce.<yourdomain> subdomain. These are independent from any existing root-domain SPF/DKIM records you may use for another SMTP provider.

5. Verify your destination addresses

The Forward action uses EmailMessage.forward(), which requires each destination to be in your account's Cloudflare Email Routing → Destination Addresses list.

  1. Dashboard → Email → Email Routing → Destination addresses
  2. Click Add destination address and enter the address (e.g. your real Gmail)
  3. Cloudflare emails a confirmation link; click it

6. Configure Cloudflare Access on /manage

Create an Access application whose policy covers cutout.<yourdomain>/manage/*. Only /manage should be protected.

Set CF_ACCESS_TEAM (your Access team slug) and CF_ACCESS_AUD (the application AUD tag) in wrangler.toml.

7. Deploy

wrangler deploy

Attach the worker's HTTP handler to whichever hostname you want /manage reachable on (Workers → your worker → Triggers → Custom Domain).

8. Write rules

Visit https://<your-cutout-host>/manage (authenticating via Access). Add Forward rules using glob patterns for the local and domain parts, e.g. *@example.com → me@gmail.com. First match wins; a *@* Drop catch-all is pinned at the bottom. Destinations must already exist in the Destination Addresses list (step 5) — otherwise forward() bounces at runtime.
