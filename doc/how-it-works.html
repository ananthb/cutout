How It Works

Cutout sits between the outside world and your real email address, acting as an invisible relay.

Inbound flow (someone emails one of your addresses)

  1. An email arrives at an address on any domain pointed at Cutout (e.g. newsletter@yourdomain.com)
  2. Cloudflare Email Routing's catch-all rule triggers the Cutout worker
  3. The worker matches the recipient against your rules, top-to-bottom
  4. On a Forward match, the worker generates one reverse-alias reply+<uuid>@yourdomain.com and saves its mapping reply+<uuid> → original sender in Cloudflare D1 (durable storage). This alias is shared across all destinations on the rule, so any reply lands back on the same sender.
  5. The worker then fans the mail out to each destination. For email destinations, there are two modes:
    • Native mode (default) → uses EmailMessage.forward(). Original bytes (PGP, attachments, formatting) pass through untouched. The Reply-To is overlaid, but may be ignored by some mail clients if the original message already has one.
    • Proxy via rewrite mode → uses Email Service structured send. Reconstructs the message to ensure Reply-To works when replying via the same custom domain, but strips PGP/MIME signatures and attachments.
    In both modes, Cutout injects an X-Original-From header containing the real sender's address for your permanent archival.
  6. Other destination kinds:
    • Telegram destinations → sendMessage with the subject + body. Cutout saves a reply context in D1 keyed by tg:<chat_id>:<message_id>.
    • Discord destinations → createMessage with a Reply button. Reply context keyed by dc:<channel_id>:<message_id>.

Reply flows

Email replies

  1. You reply from your real mailbox; the reply is addressed to the reply+<uuid>@yourdomain.com that Cutout set as Reply-To on the forwarded message
  2. Email Routing triggers the worker again
  3. The worker looks up reply+<uuid> in D1 to find the original sender
  4. The reply is sent via Email Service's structured send_email API, with the alias as the From and the original sender as the To

Bot replies

  1. You reply to a bot message in Telegram or Discord
  2. Telegram sends a webhook; Discord sends an interaction event. Both contain the message_id of the message you're replying to
  3. The worker looks up the message_id in D1 to find the original sender and target alias
  4. The reply is sent via send_email, just like an email reply

Destination verification

Email destinations must exist in your account's Cloudflare Email Routing → Destination addresses list. Cloudflare enforces this at the API level for both forward() and send_email().

Telegram and Discord destinations don't need this — they're validated via whether their bot secrets are set.

Multi-domain support

You can point any number of Cloudflare zones (domains) at the same Cutout worker; rules use the domain glob to differentiate them. Reverse-aliases are derived from the domain the mail actually arrived on.

Key components

  • Cloudflare Email Routing — inbound catch-all → worker; EmailMessage.forward() for Native mode forwarding; Destination Addresses list for verification
  • Cloudflare Email Service — used for the reverse-alias reply path, Proxy mode forwarding, and fan-out beyond the first destination
  • Cloudflare D1 — durable storage for reverse-alias mappings and bot reply contexts. Unlike KV, entries here do not expire, ensuring old email threads remain replyable forever.
  • Cloudflare KV — stores the routing rule list managed via /manage.
  • Cloudflare Access — protects /manage; /health, /telegram/webhook, /discord/interactions are public (the last two authenticate by secret token and Ed25519 signature respectively)
  • botrelay-rs — reusable Rust crate: Telegram + Discord bot clients and reply-context types, shared with sibling projects
