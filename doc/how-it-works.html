How It Works

Cutout sits between the outside world and your real email address, acting as an invisible relay.

Routing model

You configure an ordered list of rules at /manage. Each rule matches inbound mail with glob patterns on the local part and the domain (e.g. shop+*@example.com), and has an action:

  • Forward to one or more destinations (mixed kinds allowed), or
  • Drop silently.

Rules are evaluated top-to-bottom; the first match wins. A *@* catch-all is pinned to the end and defaults to Drop, so mail you haven't written a rule for is discarded rather than accidentally leaked.

Destination kinds

A Forward rule carries a list of destinations. Each is one of:

  • email:you@example.com — forwarded with EmailMessage.forward()
  • telegram:<chat_id> — posted to a Telegram chat via bot sendMessage
  • discord:<channel_id> — posted to a Discord channel via bot createMessage with a Reply button

Telegram and Discord destination kinds are only available when their bot secrets are configured (see the setup guide). The /manage UI hides the kinds that aren't currently usable.

Inbound flow (someone emails one of your addresses)

  1. An email arrives at an address on any domain pointed at Cutout (e.g. newsletter@yourdomain.com)
  2. Cloudflare Email Routing's catch-all rule triggers the Cutout worker
  3. The worker matches the recipient against your rules, top-to-bottom
  4. On a Forward match, the worker generates one reverse-alias reply+<uuid>@yourdomain.com and saves its mapping reply+<uuid> → original sender in KV (30-day TTL). This alias is shared across all destinations on the rule, so any reply lands back on the same sender.
  5. The worker then fans the mail out to each destination:
    • First email destination → EmailMessage.forward(destination, {Reply-To: reply+<uuid>}). Original bytes pass through untouched.
    • Further email destinations → Email Service structured send (Cloudflare's forward() is single-destination per message). These get rewritten headers, not the native-forwarder look.
    • Telegram destinations → sendMessage with the subject + body. Cutout saves a reply context in KV keyed by tg:<chat_id>:<message_id>.
    • Discord destinations → createMessage with a Reply button. Reply context keyed by dc:<channel_id>:<message_id>.

Reply flows

Email replies

  1. You reply from your real mailbox; the reply is addressed to the reply+<uuid>@yourdomain.com that Cutout set as Reply-To on the forwarded message
  2. Email Routing triggers the worker again
  3. The worker looks up reply+<uuid> in KV to find the original sender
  4. The reply is sent via Email Service's structured send_email API, with the alias as the From and the original sender as the To

Telegram replies

  1. In Telegram you tap Reply on the forwarded message and type
  2. Telegram POSTs an Update to /telegram/webhook
  3. The worker looks up tg:<chat_id>:<reply_to.message_id> in KV — this is the reply context saved at forward time
  4. Cutout sends an email via Email Service from the alias to the original sender, with the reply text as the body

Discord replies

  1. You click the Reply button on the bot's message in Discord
  2. Discord POSTs a component-interaction to /discord/interactions (Ed25519-verified); the worker responds with a modal
  3. You submit the modal; Discord POSTs a modal-submit interaction. The worker extracts the reply text, looks up dc:<channel_id>:<message_id> in KV, and sends an email via Email Service back to the original sender
  4. The worker responds to the interaction with an ephemeral "Reply sent." confirmation

Rule validation and tester

Before saving, cutout runs a validator over the whole rule set. Errors (block save) cover empty patterns, empty destination lists, duplicate patterns, and chat destinations whose bot secrets aren't configured. Unreachable rules — ones whose pattern is completely covered by an earlier rule — are surfaced as warnings via glob subsumption (handles * and ?).

The /manage/test page runs the current rule set against an inbound recipient you type in and shows which rule fires.

Destination verification

Email destinations must exist in your account's Cloudflare Email Routing → Destination Addresses list. Add a destination in the Cloudflare dashboard; Cloudflare sends its own confirmation email with a one-time link. Once verified there, EmailMessage.forward() will deliver to it. Telegram and Discord destinations don't need this — they're validated via whether their bot secrets are set.

Key components

  • Cloudflare Email Routing — inbound catch-all → worker; EmailMessage.forward() for the Forward action; Destination Addresses list for verification
  • Cloudflare Email Service — used only on the reverse-alias reply path and for non-primary email destinations
  • KV — rule list; reverse-alias mappings (30-day TTL); bot reply contexts (30-day TTL)
  • Cloudflare Access — protects /manage; /health, /telegram/webhook, /discord/interactions are public (the last two authenticate by secret token and Ed25519 signature respectively)
  • botrelay-rs — reusable Rust crate: Telegram + Discord bot clients and reply-context types, shared with sibling projects

Multiple domains

One worker can serve any number of zones. For each email domain, enable Email Routing and point its catch-all at the same Cutout worker; rules use the domain glob to differentiate them. Reverse-aliases are derived from the domain the mail actually arrived on.
